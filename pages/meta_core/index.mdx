# meta_core

Server-authoritative framework core for Meta Framework.

## What is meta_core?

meta_core is a **server-only** resource (`server_only 'yes'`) that owns all authoritative state. It is the single source of truth for player data in your server.

No client-side code exists in meta_core. All interactions happen through the `meta.player` API and server exports.

## API

All player operations go through `meta.player`:

```lua
-- Character data (the RP character)
meta.player.getChar(source, 'firstname')
meta.player.setChar(source, 'job', jobData)

-- Account data (the real player)
meta.player.getAcct(source, 'license')
meta.player.acctId(source)

-- State (persisted key/value)
meta.player.getState(source, 'isDead')
meta.player.setState(source, 'isDead', true)

-- Permissions
meta.player.hasJob(source, 'police', 2)
meta.player.isAdmin(source)

-- Queries
meta.player.exists(source)
meta.player.findWithJob('police')
```

## Modules

| Module | Responsibility |
|--------|---------------|
| **Player** | Player lifecycle — connection, disconnection, registry lookup |
| **Characters** | Multicharacter persistence — create, delete, select, save to MySQL |
| **Permissions** | ACE-based admin levels with role, job, and gang checks |
| **State** | Per-player key/value state bag with dirty tracking and MySQL persistence |
| **Validation** | Source verification and schema-based payload validation |

## Architecture

meta_core exposes server exports that other resources call to interact with player data. It fires events when state changes, which downstream resources (like meta_logic) can listen to and relay to clients.

```
[Your Resource] ──exports──> [meta_core] ──events──> [meta_logic server] ──push──> [client]
```

meta_core never talks to clients directly. That's meta_logic's job.
